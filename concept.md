# Deductive Design -- 演繹的デザイン

**断定の連鎖から、連続的な探索へ。AI時代の設計フレームワーク**

---

## 問いの核心

Generative Designは、AIやアルゴリズムによって大量の設計案を生成することを可能にした。しかし、根本的な問いに答えを出せていない：

> **何を根拠に方向を定め、何を根拠に選ぶのか？**

現実には、方向づけも選択も、設計者の「経験」と「直感」に委ねられている。しかし問題の根はもっと深い。従来の設計プロセスには、**「断定しなければ先に進めない」**という構造的制約がある。

---

## 従来のプロセス：断定の連鎖

### 各段階で起きている「断定」

従来の設計プロセスは、各段階で**何かを断定的にピックアップ**しないと次に進めない構造を持っている。

```
1. リサーチ
   膨大な調査結果の中から、経験や合意に基づいて
   「これが重要だ」とピックアップする ← 断定①

2. 案づくり
   ピックアップした内容を元に、A案・B案を作る
   A案とB案は離散的な選択肢として提示される ← 断定②

3. 評価
   「A案かB案か」を選ぶ ← 断定③
```

**ここで見落とされていること：**

リサーチの段階で「これが重要だ」と断定した瞬間に、他の可能性は捨てられる。案づくりの段階でA案とB案を切り出した瞬間に、AとBの間にある無数のグラデーションは消える。

本来、A案とB案は**断絶的な二者択一ではなく、連続的な空間の中の二つの点**である。その間には無数の中間形態がある。しかし、従来のプロセスでは案を作るコストが高いため、離散的な数案に絞らざるを得なかった。

```
本来の設計空間（連続的）:

  A ─────────────────────────────────── B
  ↑                                     ↑
  ここだけ        この間がすべて         ここだけ
  切り出される    見えなくなる           切り出される
```

### 断定が帰納的である理由

各段階での断定は、何に基づいて行われるか。**設計者の経験**である。

- リサーチから何をピックアップするかは、過去の経験が決める
- A案のような方向を思いつくのは、類似プロジェクトの経験があるから
- A案とB案のどちらが良いかを判断するのは、蓄積された成功・失敗体験

つまり、個別の経験（特殊）から一般的な判断を導くという帰納的な構造が、各断定の背後にある。

## デダクティブ・デザイン：断定から連続へ

### 生成AIが変えた前提

従来のプロセスで断定が必要だった最大の理由は、**案を作るコストが高かった**からである。1案作るのに時間がかかるから、3〜5案に絞り込む必要があった。リサーチの結果もすべて反映できないから、重要そうなものだけをピックアップする必要があった。

生成AIはこの前提を覆す：

| 従来 | AI時代 |
|------|--------|
| 1案作るのに時間がかかる | 即座に多数の案を生成可能 |
| 3〜5案に断定的に絞る必要 | 連続的なグラデーションを探索可能 |
| A案かB案かの二択 | AとBの間を連続的に変化させて確認可能 |
| 条件変更 → 作り直し | 条件変更 → 即座に反映 |
| リサーチ結果のピックアップが必要 | すべての条件を同時に反映可能 |

### 「断定」から「連続的な探索」へ

デダクティブ・デザインの核心は、各段階での断定を**連続的な探索**に置き換えることにある。

**リサーチ → 案づくりの断定を解消：**
リサーチの結果から「これが重要」と一つに絞るのではなく、すべての条件を生成のパラメータとして保持する。「環境条件を重視するとこう、文脈を重視するとこう」という連続的なスペクトルの中で案を確認できる。

**A案・B案の断絶を解消：**
離散的な2案ではなく、AとBの間を連続的に変化させた設計空間を提示できる。「Aに近いがBの要素を少し取り入れた案」「AとBのちょうど中間」といったグラデーションが可視化される。

```
従来: A ─── (見えない) ─── B    → AかBかを断定

AI時代: A ── a1 ── a2 ── ab ── b1 ── b2 ── B    → 連続的に探索
```

**評価の断定を解消：**
「A案かB案か」ではなく、連続的なスペクトルの中で「この方向に寄せていく」「この条件をもう少し強めると」という段階的な収束が可能になる。

### なぜ「演繹的」なのか

従来の帰納的プロセスでは、設計者が**経験に基づいて断定する**ことで次に進む。この断定の根拠は個人の内部にある。

デダクティブ・デザインでは、**リサーチによって得られた条件群**が設計空間を連続的に規定し、フィルタリングによって漸進的に収束していく。断定は不要になる。条件という一般的な原理から、特殊な最適解が連続的に導かれる——これが演繹法の構造をとる理由である。

```
帰納的（従来）:
  経験 → 断定① → 断定② → 断定③ → 解

演繹的（本手法）:
  リサーチ → 条件群 → 連続的な生成空間 → 漸進的フィルタリング → 解
  （各段階で断定は不要。条件に基づいて連続的に収束する）
```

## フレームワーク：連続的スパイラル

### 3つの活動の同時進行

デダクティブ・デザインは、離散的な「フェーズ」ではなく、3つの活動が連続的に螺旋を描きながら進む**スパイラルプロセス**として構成される。

```
     RESEARCH ──→ GENERATE ──→ FILTER
         ↑                        │
         │    新たな問いの発見      │
         └────────────────────────┘

     この循環が、解像度を上げながら繰り返される
     断定ではなく、漸進的な収束
     最終的にDISTILL（人間の判断を集中投入）
```

### RESEARCH -- リサーチ

すべての起点であり、プロセス全体を通じて継続する活動。

**リサーチの役割の変化：**
従来のリサーチは「調べた結果から重要なものを断定的にピックアップする」行為だった。デダクティブ・デザインでは、リサーチの結果は**すべてが条件として保持され、生成とフィルタリングに連続的に反映される**。「これが重要」と断定する必要はない。

リサーチの7領域：

| 領域 | 問い |
|------|------|
| 環境 | この土地の環境特性は何か |
| 法規・制度 | 何が許され、何が促進されるか |
| 文化・歴史 | この場所の物語は何か |
| 経済 | 何が実現可能か |
| 社会 | 誰のために、何が求められているか |
| 技術 | 何が技術的に可能か |
| 先行事例 | 同様の試みから何を学べるか |

**生成された案がリサーチを誘発する：** 案を見ることで初めて「ここをもっと調べるべきだ」と気づく。この循環がスパイラルを駆動する。

### GENERATE -- 生成

リサーチで得た条件に基づき、**連続的な設計空間**を生成する。

従来との決定的な違い：
- 離散的なA案・B案ではなく、**パラメータ空間の中の連続的なバリエーション**を生成する
- 「AとBの間」も含めた全体のグラデーションが見える
- 条件の重み付けを変えることで、スペクトル上を自在に移動できる
- リサーチと同時進行し、発見が即座に生成に反映される

### FILTER -- フィルタリング

リサーチで構築した条件群により、設計空間を**漸進的に**収束させる。

従来のフィルタリングとの違い：
- 「この案は残す／捨てる」の二値判定ではなく、条件への適合度のグラデーションとして評価
- 条件が追加されるたびに設計空間が連続的に狭まる
- フィルタリングの過程で条件自体の妥当性も検証される

**フィルタリングの原則：**
- 各フィルタは「経験的好み」ではなく「リサーチに基づく条件」で構成される
- フィルタの基準は明示され、チームで共有される
- 過程と結果はすべて記録され、検証可能である

### DISTILL -- 最終蒸留

スパイラルを経て十分に収束した段階（3〜10案）で、人間の判断を集中投入する。

ここで初めて、設計者の感性・直感・美学的判断が入る。ただしこれは「180案から勘で選ぶ」のではなく、**連続的なフィルタリングを経て十分に収束した候補群に対して、人間の判断力を最も有効に使う**行為である。

## 連続性がもたらすもの：合意形成の変容

### 従来の合意形成

```
設計者: 「A案とB案があります。どちらにしますか？」
クライアント: 「A案のこの部分は好きだが、B案のこの要素も欲しい」
設計者: 「ではC案を作り直します」（大きな手戻り）
```

A案とB案が断絶的であるため、「その間」を議論することが難しい。クライアントも「AかBか」の二択を迫られ、本当に欲しいものを表現しにくい。

### デダクティブ・デザインでの合意形成

```
設計者: 「この条件下で、設計空間はこのように広がっています」
        「環境性能を重視するとこちら方向、文脈を重視するとこちら方向です」
クライアント: 「環境も大事だが、文脈ももう少し...」
設計者: 「では、このあたり（連続的な中間点）を見てみましょう」
        （即座に生成して確認）
```

連続的な設計空間の中で、**「もう少しこちらに」「この条件をもう少し強く」**といった漸進的な対話が可能になる。これにより：

- クライアントは二択ではなく、方向性を表現できる
- 設計者とクライアントのイメージのすり合わせが容易になる
- 「作り直し」ではなく「調整」で済む
- 合意の過程自体が、条件に基づいた透明なものになる

## コンセプトの本質：「目利き」の外在化

熟練した建築家の「目利き」は、実は膨大な経験を通じて内面化されたリサーチの蓄積である。彼らは無意識のうちに、連続的な設計空間を頭の中に持ち、多角的な条件で候補を評価している。

デダクティブ・デザインは、この**暗黙のプロセスを外在化し、体系化する**。

- 「断定」として隠されていた判断を、条件として明示化する
- 「離散的な案」として切り出されていた設計空間を、連続的に可視化する
- 「経験」として個人に閉じていた知識を、リサーチとして共有可能にする

生成AIは、この外在化を技術的に可能にした媒介である。

## Generative Designとの関係

デダクティブ・デザインはGenerative Designを**前提とし、その先を完成させる**。

| 側面 | Generative Design | Deductive Design |
|------|-------------------|-----------------|
| 問い | いかに多くの可能性を生み出すか | 何を根拠に方向づけ、選び取るか |
| 焦点 | 生成の量と多様性 | リサーチに基づく連続的な収束 |
| 設計空間 | 大量の離散的候補 | 連続的なパラメータ空間 |
| プロセス | 制約定義 → 自動生成 | リサーチ ↔ 生成 ↔ フィルタリング（同時進行） |
| 各段階の接続 | 断定による離散的遷移 | 条件による連続的収束 |

## 評価基準フレームワーク

フィルタリングに用いる評価軸（プロジェクトに応じて重み付けを調整）：

| 評価軸 | サブ指標 | リサーチ手法 |
|--------|---------|-------------|
| **性能** | 構造安全性 / 環境性能 / 耐久性 / 機能充足率 | シミュレーション |
| **経済** | 初期コスト / ランニングコスト / 投資回収期間 | コスト調査 |
| **文脈** | 地域整合性 / 景観調和 / 歴史的連続性 | フィールドリサーチ |
| **体験** | 空間の質 / 動線 / 光と影 / 素材感 | VR/AR体験評価 |
| **社会** | 環境負荷 / アクセシビリティ / コミュニティ | 社会調査 |
| **美学** | プロポーション / 構成 / 独自性 / 時間的耐性 | 専門的評価 |

### 意思決定原則

1. **透明性**: 判断基準と重み付けを事前に明示する
2. **外在性**: 選択の根拠を調査可能な外部条件に求める
3. **連続性**: 断定的な二択ではなく、連続的な設計空間の中で漸進的に収束する
4. **同時性**: リサーチ・生成・フィルタリングを分離せず、連続的に回す
5. **人間中心**: 最終判断は人間が行う。十分に収束した候補に対して
